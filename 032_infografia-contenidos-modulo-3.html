3.1

Capítulo 3.1 · #tema/frameworks

Frameworks de Deep Learning

¿Por qué no programar la red a mano? Implementar la retropropagación y la diferenciación automática en NumPy es inviable. Un framework aporta autograd, capas listas, optimizadores, soporte de GPU y un ecosistema de despliegue. El módulo recorre los dos dominantes —TensorFlow/Keras y PyTorch— y la convergencia reciente que habilita Keras 3 como puente multi-backend.

TensorFlow + Keras: las tres puertas para construir un modelo

Keras es la interfaz de alto nivel de TensorFlow («para seres humanos, no para máquinas»). Ofrece tres APIs con creciente flexibilidad y creciente complejidad: se elige según la forma del modelo.

sequential.py Python · Keras
# API Sequential — la línea de ensamblaje modelo = keras.Sequential([ keras.layers.Dense(128, activation='relu'), keras.layers.Dropout(0.3), keras.layers.Dense(1, activation='sigmoid'), ]) modelo.compile(optimizer='adam', loss='binary_crossentropy', metrics=['accuracy'])

La línea de producción de café: en Sequential el grano pasa de una etapa a la siguiente sin desviarse (una entrada, una salida). El Funcional es el acueducto de Bogotá: el agua se bifurca y se une por caminos paralelos. El Subclassing es el artesano que fabrica cada pieza a mano: máxima libertad, máximo cuidado.

PyTorch: tensores, grafo dinámico y nn.Module

PyTorch construye el grafo de cómputo en tiempo real, operación por operación, a medida que el código se ejecuta (define-by-run). Esto permite usar if, for y depurar con print como en Python normal. requires_grad=True activa el seguimiento de gradientes.

model.py Python · PyTorch
# Todo modelo hereda de torch.nn.Module class MiRed(nn.Module): def __init__(self): super().__init__() self.fc1 = nn.Linear(4, 64) self.fc2 = nn.Linear(64, 1) def forward(self, x): # grafo dinámico x = torch.relu(self.fc1(x)) return torch.sigmoid(self.fc2(x))

El río Magdalena: un grafo estático (TF 1.x) era un canal de concreto fijo; el grafo dinámico de PyTorch es el río Magdalena — su cauce se adapta al terreno, se bifurca y crea meandros según el código que corre.

Control explícito vs. abstracción: en Keras, model.fit() encapsula todo el ciclo. En PyTorch el bucle de entrenamiento se escribe a mano, lo que da control fino sobre cada paso (clave para precisión mixta y acumulación de gradientes).

Keras 3: un código, tres motores (multi-backend)

Lanzado a finales de 2023, Keras 3 es una reescritura que convierte a Keras en un framework multi-backend: el mismo código de alto nivel corre sobre TensorFlow, PyTorch o JAX (y OpenVINO para inferencia) cambiando una sola variable de entorno.

backend.py Python · Keras 3
# Elegir el motor ANTES de importar keras import os os.environ['KERAS_BACKEND'] = 'jax' # o 'tensorflow' / 'torch' import keras # el mismo código corre en los 3 motores

Capítulo 3.2 · #tema/gpu · #tema/hpc

Computación de Alto Rendimiento (GPU)

Una red profunda son miles de millones de multiplicaciones de matrices. La CPU las hace de forma casi secuencial; la GPU las hace en paralelo masivo. Esa es la diferencia entre entrenar en semanas y entrenar en horas. Este capítulo democratiza el HPC: con Colab y Kaggle, cualquier estudiante con una cuenta de Google accede a una GPU.

CPU vs. GPU: pocos núcleos potentes vs. malla de núcleos simples

La CPU tiene pocos núcleos muy rápidos y versátiles; la GPU tiene miles de núcleos simples que ejecutan la misma operación sobre muchos datos a la vez (SIMD). Para álgebra de tensores, gana la malla.

La despulpadora y la panadería: despulpar a mano (CPU) es preciso pero lento; la despulpadora industrial (GPU) procesa cientos de granos a la vez. La CPU es como un maestro panadero versátil; la GPU, como miles de operarios haciendo la misma tarea simple en paralelo.

Núcleos CUDA vs. Tensor Cores

Los Tensor Cores (desde Volta, 2017) son unidades especializadas en multiplicación-acumulación de matrices: un solo Tensor Core ejecuta una multiplicación \(4\times 4\) en una instrucción. En una A100, los CUDA dan 19,5 TFLOPS en FP32 y los Tensor Cores 312 TFLOPS en FP16 (≈16× más).

$$\text{Tensor Core: } C \mathrel{+}= A_{4\times4}\cdot B_{4\times4}\ \text{(1 instrucción)}$$

Magia transparente: al ejecutar model.fit() en una T4 de Colab, detrás se activan CUDA, cuDNN y los Tensor Cores sin escribir una línea de bajo nivel.

VRAM: el recurso más escaso

La VRAM almacena a la vez los pesos, los estados del optimizador, los gradientes, las activaciones y el lote. Los estados del optimizador (Adam) suelen ser el mayor consumidor.

$$\text{VRAM}_{\text{pesos}} \approx 2N\ \text{GB}\quad(N \text{ mil millones, FP16})$$

La bodega de la plaza: no importa cuántos camiones (datos) lleguen ni cuántos vendedores (núcleos) haya — si la bodega (VRAM) se llena, no entra más mercancía. Ese es el temido CUDA OOM.

Precisión mixta (AMP): hacer más con menos

Representar las activaciones en 16 bits en vez de 32 reduce la memoria de esos tensores a la mitad y acelera el cómputo al activar los Tensor Cores. Para estabilidad, AMP mantiene una copia de los pesos maestros en FP32.

Matiz: el ahorro se da sobre todo en activaciones; el ahorro total es menor que la mitad porque AMP guarda los pesos maestros en FP32 para el optimizador.

FP16 vs bfloat16: en GPUs Ampere+ (L4/A100) suele preferirse bfloat16 (sin GradScaler); en la T4 se usa float16 con escalado de pérdida.

amp_train.py Python · PyTorch AMP
# Entrenamiento con precisión mixta (PyTorch AMP) scaler = torch.amp.GradScaler() for X, y in loader: optim.zero_grad() with torch.autocast("cuda", dtype=torch.float16): # cómputo en FP16 perdida = criterio(modelo(X), y) scaler.scale(perdida).backward() # escala para evitar underflow scaler.step(optim); scaler.update()

Multi-GPU: DP, DDP y FSDP

Cuando una GPU no basta, se reparte el trabajo entre varias.

  • ·DataParallel (DP): simple pero ineficiente — cuello de botella en la GPU-0.
  • ·DistributedDataParallel (DDP): un proceso por GPU, sincroniza con NCCL (≈3× más utilización). La opción recomendada.
  • ·FSDP (Fully Sharded): fragmenta pesos, gradientes y estados del optimizador entre GPUs — permite entrenar modelos que no caben en una sola.

En TensorFlow, el equivalente es tf.distribute.MirroredStrategy (un dispositivo, varias réplicas).

Pipelines de Datos y Despliegue

El restaurante eficiente: tener la mejor GPU (cocina) no sirve si los platos llegan despacio. El pipeline de datos es la logística — preparar, refrigerar, mezclar y servir los lotes al ritmo de la cocina, sin que esta espere. Y un modelo entrenado que nadie puede usar tampoco sirve: el despliegue lo lleva a la tablet en el cafetal.

El orden del pipeline tf.data importa

El orden de las operaciones cambia la eficiencia. La regla práctica: map → cache → shuffle → batch → prefetch, con prefetch(AUTOTUNE) siempre al final.

La cocina del restaurante: primero se prepara el ingrediente (map), se refrigera lo costoso (cache), se baraja el menú (shuffle), se arman bandejas (batch) y un mesero adelanta la siguiente bandeja (prefetch) para que la cocina nunca pare.

Prefetch: solapar CPU y GPU

Sin prefetch, la GPU queda ociosa mientras la CPU prepara el siguiente lote. Con prefetch, ambos trabajan en paralelo y el tiempo por época se acerca al mayor de los dos, no a su suma.

$$t_{\text{época}} \approx \max\!\left(t_{\text{CPU}},\; t_{\text{GPU}}\right)$$

En tf.data: .prefetch(AUTOTUNE). En PyTorch: num_workers, pin_memory y prefetch_factor en el DataLoader.

num_workers = meseros en paralelo: varios procesos preparan lotes a la vez. Demasiados, sin embargo, saturan la CPU y la RAM.

pipeline.py Python · tf.data + PyTorch
# El mismo pipeline en ambos frameworks ds = (ds.map(carga, num_parallel_calls=AUTOTUNE) # tf.data .cache().shuffle(1_000) .batch(32).prefetch(AUTOTUNE)) loader = DataLoader(dataset, batch_size=32, # PyTorch num_workers=4, pin_memory=True, prefetch_factor=2)

Data Augmentation: entrenar en condiciones variadas

Multiplicar artificialmente el dataset con transformaciones reduce el sobreajuste — solo si respetan la semántica. Una transformación que cambia la etiqueta real hace más daño que bien.

El director técnico: entrena al equipo bajo lluvia, sol y cancha mojada (variar condiciones) para que rinda en cualquier partido — pero no le cambia las reglas del juego. Aplicar ≥4 transformaciones y comparar el accuracy de validación con y sin aumento.

Guardado y exportación: cada framework, su formato

.pth sale de PyTorch y .keras de Keras: son dos implementaciones equivalentes, no el mismo objeto en dos archivos. ONNX es el formato abierto que las hace portables entre runtimes.

Formato Origen Uso típico

ONNX, el pasaporte universal: con él un modelo cruza fronteras de framework y de hardware. Pero siempre hay que verificar la consistencia de predicciones tras exportar: un modelo que predice distinto no es desplegable.

Flujo de despliegue: del notebook al campo

La rama visual se cuantiza y corre con LiteRT offline en la tablet; la rama temporal se sirve por API REST en la nube. ONNX actúa como pasaporte interoperable.

exportar.py Python · ONNX
# Exportar PyTorch → ONNX (pasaporte portable) torch.onnx.export( modelo, ejemplo_entrada, "modelo.onnx", input_names=["img"], output_names=["logits"], dynamic_axes={"img": {0: "batch"}})

Despliegue responsable: en campo, un falso positivo/negativo afecta decisiones reales del agricultor. La ruta explicita transparencia y mitigaciones concretas.
